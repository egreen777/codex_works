## 주식투자 백테스팅 작업 가이드

## 이 문서의 목적
- 이 문서는 사용자가 백테스팅 프로젝트에 대해 내리는 작업 지침을 정리하는 문서이다.
- 전략 아이디어와 개발 지침이 함께 출발했지만, 앞으로는 이 문서를 상위 작업 가이드로 사용한다.
- 세부 전략 정의는 `strategy.md`, 구현 기준은 `build.md`, 사용법은 `README.md`로 분리한다.
- 이 문서는 이후 작업 방향을 결정하는 기준 문서로 유지한다.

## 관련 md파일의 역할
1. backtester : python기반의 backtesting tool 개발 
- `backtester/README.md`
  - 사용자가 기능을 실행할 때 보는 사용 가이드
  - 실행 방법, 옵션, 출력 결과를 설명
- `backtester/strategy.md`
  - 투자 전략의 상세 규칙을 정리
  - 전략별 판단 규칙, 투자 대상, 리밸런싱 규칙을 관리
- `backtester/build.md`
  - 구현 지침을 정리
  - 데이터 로딩 방식, CLI 규칙, 출력 형식, 로그 규칙, 일관성 유지 규칙을 관리
- `invest_backtest.md`
  - 사용자가 작업 지침을 정리하는 상위 가이드
  - 전략 방향과 개발 방향을 함께 지시하는 기준 문서

2. tradingview : backtester에서 만든 전략을 TradingView 기반의 pine script로 개발 
- 주의: backtester/strategy.md 파일을 참고하여 만들며, strategy.md 파일은 수정하면 안됨.
- `tradingview/README.md`
  - 사용자가 기능을 실행할 때 보는 사용 가이드
  - 실행 방법, 옵션, 출력 결과를 설명
- `tradingview/strategy.md`
  - backtester/strategy.md 파일을 기반. Tradingview환경에 맞춘 투자전략의 상세 규칙을 정리 
  - 전략별 판단 규칙, 투자 대상, 리밸런싱 규칙을 관리
  - Python 백테스터와 다른 점은 `[Pine script 차이점]` 섹션에 정리
- `tradingview/pine_build.md`
  - tradingview/strategy.md 에서 정의한 투자전략에 대한 Tradingview환경에 맞춘 구현 지침을 정리

## 전략수립 가이드라인
1. 목표를 먼저 정하기
  - 단타/스윙/장기투자 중 무엇인지: 장기
  - 국내주식/미국주식/ETF 중 무엇인지: 미국 ETF
  - 일봉 기준인지, 분봉까지 할 건지: 일봉
  - 단순 전략 검증용인지, 실전매매 보조용인지: 전략검증

2. 백테스트 엔진 관점에서 필요한 전략 요소
  - 핵심은 과거 데이터에 대해 매수/매도 규칙을 순서대로 적용하는 것
  - 가격 데이터 로딩: 필요
  - 전략 신호 생성: 필요
  - 주문 체결 시뮬레이션: 필요
  - 수수료/세금 반영: nice to have이며 첫번째 버전에서는 필요없음
  - 포트폴리오 가치 계산: 필요

3. 데이터 품질 관점
  - 수정주가 반영 여부: 필요
  - 액면분할/배당 처리: 필요
  - 상장폐지 종목 포함 여부: 필요없음
  - 생존자 편향 방지: 필요없음
  - 분봉이면 틱/체결 지연도 고려: 필요없음

4. 검증 지표
  - 누적수익률: 필요
  - 연복리수익률(CAGR): 필요
  - MDD: 필요
  - 승률: 필요없음
  - 손익비: 필요없음
  - 샤프지수: 필요없음
  - 거래 횟수: 필요없음
  - 포지션별 평균 보유기간: 필요

5. 과최적화 방지 장치
  - 학습 구간 / 검증 구간 / 테스트 구간 분리
  - 워크포워드 테스트
  - 파라미터를 너무 많이 두지 않기
  - 전략이 여러 종목, 여러 기간에서 재현되는지 보기

6. 종목후보
  1) 주식 자산
    - VOO, SPY, SSO, SPXL
    - QQQ, QLD, TQQQ
    - EEM
  2) 채권 자산
    - SHY, IEF, TLT
  3) 기타 자산
    - GLD, SLV, COPX

7. 전략후보
  1) 듀얼 모멘텀전략
    - 상반된 경향이 있는 주식 자산을 2개 선정
    - 예: EEM 과 VOO 또는 VOO 와 QQQ
    - 적정기간 (120일 200일)의 수익율이 채권보다 높을 경우 높은 자산 1개에 투자
    - 2개 자산 모두 채권보다 낮을 경우에는 채권 보유
    - 주식자산과 채권 모두 마이너스라면 현금을 보유
    - 매달말 또는 지정 주기 말에 한번씩 리밸런싱
  2) 추세추종 전략
    - 50일, 120일, 200일 등 주요 이동평균선의 추세 상태를 기준으로 판단
    - 상승 추세면 매수, 이탈 시 매도
  3) 자산배분 전략
    - 주식자산 중 1개, 채권자산 중 1개, 기타자산 중 1개를 선정하여 각각 비율로 매수
    - 적정기간 또는 연말 기준으로 비중이 늘어난 곳을 줄이고 줄어든 곳을 늘려 처음 비율로 복원

## 개발 가이드라인
  1) 프로젝트 기본 방향
      - `backtester/` 폴더 아래에서 개발
      - Python 기반 CLI 도구로 구현

  2) 데이터 입력 형식
      - CSV 기반 일봉 데이터 사용
      - CSV 형식은 `Date, Open, High, Low, Close, Adj Close, Volume`

  3) 가격 데이터 수집
      - 주식가격 정보(CSV)는 외부 검색을 통해 구함
      - 주로 Yahoo Finance, Google Finance, Investing.com 등을 후보로 검토
      - 적절한 source를 확인하고 선정
      - 구해진 가격 정보는 `backtester/data/` 폴더 아래에 넣을 것
      - 해당 ticker의 파일이 있고 필요한 기간을 모두 포함하면 기존 파일 사용
      - 없는 경우에만 새로 정보를 구할 것

  4) 전략/종목 확장 규칙
      - 종목은 `종목후보`에 있는 종목을 사용
      - 종목을 추가할 경우 `종목후보`에 추가로 작성
      - 전략은 `전략후보`에 있는 전략을 사용
      - 전략을 추가할 경우 `전략후보`에 추가로 작성

  5) 전략 문서화 규칙
      - 전략의 상세 내용은 `backtester/strategy.md`에 작성
      - 여러 전략이 있으므로 각각 구분하여 정리

  6) CLI 구현 규칙
      - 종목 선택 및 기간 설정 가능해야 함
      - 투자금액 설정 가능해야 함
      - 초기 투자금은 필수
      - 매년 연초 추가 투자금은 선택
      - 투자자산이 1개가 아닌 경우:
        - 비율이 설정되어 있으면 그 비율만큼 매수
        - 비율이 설정되어 있지 않으면 동일비중 또는 남는 비중을 배분

  7) 파라미터 규칙
      - `trading-period`
        - 트레이딩 데이 기준 판단 기간
        - 추세추종에서는 이동평균선 기간
        - 듀얼 모멘텀과 자산배분에서는 판단용 window period
      - `rebalancing-period`
        - 캘린더 기준 리밸런싱 주기
        - `1m`, `3m`, `6m`, `9m`, `1y`
        - 실제 실행일은 각 기간의 마지막 거래일
      - 전략과 맞지 않는 파라미터는 무시 가능
      - 무시 시 사용자에게 ignored 메시지를 보여줄 것

  8) 성과표 출력 규칙
      - 타이틀을 붙일 것
      - 예: `MY BACKTESTING REPORT`
      - 사용한 전략 이름과 상세내용을 보여줄 것
      - 매해 성과를 표시
      - 최종 성과도 표시
      - 벤치마크 성과도 같이 표시
      - 벤치마크 규칙:
        - 종목이 1개인 경우 그 종목이 벤치마크
        - 종목이 N개인 경우 `VOO`가 벤치마크
      - 표 형식은 칸이 어긋나지 않도록 white space를 적절히 둘 것

  9) 로그 규칙
      - 전략 동작 확인용 로그를 남길 것
      - 위치: `backtester/output/backtest.log`
      - 리밸런싱 날짜별 판단 내용과 누적수익률을 기록할 것
      - 향후 다른 문제점을 확인하는 용도로 재사용 가능해야 함

  10) 차트
      - 차트는 나중에 구현
      - 첫 버전에서는 제외
